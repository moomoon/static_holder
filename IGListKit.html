用IGListKit构建可扩展的列表页面

UICollectionView是iOS上最常用的列表组件。它提供了一套十分灵活的API，但很多时候并不能直接完全满足我们的需求。今天我们将利用IGListKit搭建一个简单的应用，看看它能为我们的开发带来什么便利。

问题

你是一位短视频创业者，现在要开发一个视频广场页面，页面中有两列视频缩略图，每个缩略图底部带有视频作者信息。用户点击缩略图后可以跳转到对应的视频详情页。 作为一名熟练的iOS开发者，你在三十分钟内完成了这样的页面:

private var items: [Item] = [] {
  didSet {
    //有新数据时，刷新整个页面
    collectionView?.reloadData()
  }
}

override func viewDidLoad() {
  super.viewDidLoad()
  //在UICollectionView渲染之前，注册cell类型
  collectionView?.register(ItemCell.self, forCellWithReuseIdentifier: "cell")
}

//返回cell数量
override func collectionView(_ collectionView: UICollectionView, numberOfItemsInSection section: Int) -> Int {
  return items.count
}

//创建并配置cell
override func collectionView(_ collectionView: UICollectionView, cellForItemAt indexPath: IndexPath) -> UICollectionViewCell {
  let cell = collectionView.dequeueReusableCell(withReuseIdentifier: "cell", for: indexPath) as! ItemCell
  cell.set(item: items[indexPath.item])
  return cell
}
//处理用户点击
func collectionView(_ collectionView: UICollectionView, didSelectItemAt indexPath: IndexPath) {
  items[indexPath.item].video.showDetail(from: self)
}

//计算cell尺寸
func collectionView(_ collectionView: UICollectionView, layout collectionViewLayout: UICollectionViewLayout, sizeForItemAt indexPath: IndexPath) -> CGSize {
  let width = collectionView.bounds.width / 2
  return CGSize(width: width, height: width + 20)
}

这是一个很标准的列表页面，只需展示一种类型的数据。所有Cell的配置信息都可以直接从数据源中获取。当有新数据插入时，我们仅仅直接调用reloadData(), 强制刷新整个页面。

APP平稳运营了一段时间，用户量稳步上升，我们开始了第二版的开发工作。计划在广场上随机插入运营广告，并给平台上当天推荐的大V一个占满整个屏幕宽度的cell。碰巧有小伙伴提到下星期有个路演名额，但旧版本有些简陋，最好带着开发完成的新版本去展示。排来排去，广场的新需求只能在一个工作日内完成。过了两小时，类似这样的代码不停在项目里蔓延开来：

可以看到，每增加一种渲染的数据类型，都要修改多处分离的代码。很快整个项目就会变得十分难以理解和维护。怎样才能避免这样代码失控的情况出现呢？

IGListKit

IGListKit是Instagram为UICollectionView推出的数据驱动框架，让程序员能创建更快、更灵活的列表页面。在IGListKit中，我们将不再直接控制UICollectionView，而是将所有数据以Section的方式提供给ListAdapter，然后只需调用performUpdates(animated:completion:)方法，就可以完成从渲染到动画的全部工作。

让我们用IGListKit重新实现一下上面的需求:

private let dataSource: DataSource = .init()
lazy var listAdapter: ListAdapter = {
  let listAdapter = ListAdapter(updater: ListAdapterUpdater(), viewController: self)
  listAdapter.dataSource = dataSource
  return listAdapter
}()

override func viewDidLoad() {
  super.viewDidLoad()
  listAdapter.collectionView = collectionView
}

IGListKit用ListAdapter替代了UICollectionViewDataSource以及UICollectionViewDelegate，并额外定义了一个ListAdapterUpdater用于重载对UICollectionView相关刷新方法的调用方式。

private class DataSource: ListAdapterDataSource {
  var sections: [SectionItem] = []

  func objects(for listAdapter: ListAdapter) -> [ListDiffable] {
    return sections
  }

  func listAdapter(_ listAdapter: ListAdapter, sectionControllerFor object: Any) -> ListSectionController {
    return (object as? SectionControllerBounded)?.listAdapter(listAdapter, sectionControntrollerFor: self) ?? ListSectionController()
  }

  func emptyView(for listAdapter: ListAdapter) -> UIView? {
    return nil
  }
}

ListAdapterDataSource用于向ListAdapter提供数据以及ListSectionController。在这里我们可以遍历支持的数据类型并返回相应的ListSectionController，或者利用协议将创建过程分发到对应类型.

protocol SectionControllerBounded {
  func listAdapter(_ listAdapter: ListAdapter, sectionControntrollerFor dataSource: ListAdapterDataSource) -> ListSectionController
}

typealias SectionItem = ListDiffable & SectionControllerBounded

ListSectionController负责渲染列表中的一块区域，其中包含数量不定的cell，以及一些辅助或者装饰性视图。在这里，我们需要提供渲染Cell的方法，以及一些额外的辅助信息。

class ADSectionController: ListSectionController {
  var sectionItem: ADSectionItem!
  override func cellForItem(at index: Int) -> UICollectionViewCell {
    let cell = collectionContext!.dequeueReusableCell(of: ADCell.self, for: self, at: index) as! ADCell
    cell.set(item: sectionItem.item)
    return cell
  }
  override func didUpdate(to object: Any) {
    sectionItem = object as! ADSectionItem
    super.didUpdate(to: object)
  }
}

很自然地, 我们可以在ListSectionController上设置段落缩进，行间距，Cell间距等信息：

override init() {
  super.init()
  inset = UIEdgeInsets(top: 0, left: 8, bottom: 0, right: 8)
  minimumLineSpacing = 8
  minimumInteritemSpacing = 8
}

如果一个ListSectionController的view model可以一直拆分到Cell等级，我们还可以利用ListBindingSectionController简化Cell的配置过程：

class VideoSectionController: ListBindingSectionController<ChunkOfVideos>, ListBindingSectionControllerDataSource
 {
  override init() {
    super.init()
    //所有管理view model和cell的方法都由dataSource提供
    self.dataSource = self
  }

  //返回cell 的view model
  func sectionController(_ sectionController: ListBindingSectionController<ListDiffable>, viewModelsFor object: Any) -> [ListDiffable] {
    guard let chunk = object as? ChunkOfVideos else { return [] }
    return chunk.videos
  }

  //返回的cell需要继承ListBindable协议，用以绑定view model
  func sectionController(_ sectionController: ListBindingSectionController<ListDiffable>, cellForViewModel viewModel: Any, at index: Int) -> UICollectionViewCell & ListBindable {
    return collectionContext!.dequeueReusableCell(of: VideoCell.self, for: self, at: index) as! VideoCell
  }

  //计算cell大小
  func sectionController(_ sectionController: ListBindingSectionController<ListDiffable>, sizeForViewModel viewModel: Any, at index: Int) -> CGSize {
    return fixedCellSize
  }
}

在ListBindingSectionControllerDataSource中，我们只需要提供一系列view model，一个创建Cell的方法和一个返回Cell大小的方法，IGListKit就会自动管理并配置相应Section的渲染过程。值得注意的是，在普通的ListSectionController中，IGListKit只是简单地把Cell的相关创建请求和事件分发下来，而在ListBindingSectionController中，IGListKit可以知道Cell将要表达的数据本身，从而能更精确地计算列表的改变。

当我们的Section内部有额外的层级关系时，我们可以用ListStackedSectionController将这种关系反映在代码上： let sectionController = ListStackedSectionController(sectionControllers: [VideoSectionController(), ADSectionController()])

在这里，我们将一个由视频和广告组合而成的Section一分为二，这样在它们内部就不需要做额外的类型判断，提高了代码的可维护性。ListStackedSectionController在内部计算了所有子SectionController的项目数量以及在Section中的偏移量，并将所有的事件按照偏移量分发到对应的SectionController中。事实上，我们可以组合出无限多的层级。但需要注意的是，这些额外的层级只负责子视图的渲染，对于UICollectionView来说是不可见的，不能分别设置缩进等信息。

前后对比，在使用IGListKit之后，我们将数据渲染逻辑完全剥离开来，变成一系列可选插件。每次我们要添加新的渲染数据类型，只要创建相应的Cell和SectionController，并通过协议或者方法绑定数据源即可，不会对页面本身的逻辑产生任何影响。

代理

在Cocoa传统中，代理对象通常会被存储在delegate属性上，而且父类和子类常常会共用这一个属性。不论是否合理，这是苹果开发者社区中最常见的做法。 由于IGListKit要将datasource和delegate的方法都分发到对应的ListSectionController中，它必须要占用UICollectionView的Delegate属性。这时候，如果我们想截获一些UIViewController的变化事件，又不想写丑陋的KVO，该怎么办呢？ IGListKit在ListAdapter上提供了scrollViewDelegate和collectionViewDelegate这两个属性。望文生义，前者似乎能截获ScrollView相关的事件，后者应该能充当UICollectionView的代理。 但其实在这里，ListAdapter的行为非常微妙。当我们没有设置任何代理时，ListAdapter会将自己设置到UICollectionView的delegate上，而一旦我们设置了上述两个代理的任意一个，ListAdapter就会创建一个NSProxy对象，将事件分发到自身或两个代理其中之一。由于ListAdapter要保证框架的正确运行，它会强制截获一系列的事件，具体列表如下：

static BOOL isInterceptedSelector(SEL sel) {
  return (
          // UICollectionViewDelegate
          sel == @selector(collectionView:didSelectItemAtIndexPath:) ||
          sel == @selector(collectionView:willDisplayCell:forItemAtIndexPath:) ||
          sel == @selector(collectionView:didEndDisplayingCell:forItemAtIndexPath:) ||
          // UICollectionViewDelegateFlowLayout
          sel == @selector(collectionView:layout:sizeForItemAtIndexPath:) ||
          sel == @selector(collectionView:layout:insetForSectionAtIndex:) ||
          sel == @selector(collectionView:layout:minimumInteritemSpacingForSectionAtIndex:) ||
          sel == @selector(collectionView:layout:minimumLineSpacingForSectionAtIndex:) ||
          sel == @selector(collectionView:layout:referenceSizeForFooterInSection:) ||
          sel == @selector(collectionView:layout:referenceSizeForHeaderInSection:) ||
          // UIScrollViewDelegate
          sel == @selector(scrollViewDidScroll:) ||
          sel == @selector(scrollViewWillBeginDragging:) ||
          sel == @selector(scrollViewDidEndDragging:willDecelerate:) ||
          sel == @selector(scrollViewDidEndDecelerating:)
          );
}

对于以上事件，ListAdapter都会在截获之后，分别分发到section controller或者代理上。 接下来我们要将之前的大V cell扩展成一整个Section，展示大V的多个作品，Section顶部展示头像姓名，并且可以配置Section背景。 在SectionController中，header是由supplementaryViewSource来控制的，我们只需要像在UICollectionViewDataSource中一样返回一个UICollectionReusableView即可。 关于如何处理Section背景，iOS并没有默认的官方支持。但是我们可以利用UICollectionViewLayout 中的DecorationView概念，在对应的Section底部插入一个View（事实上，这是一种非常流行的做法）。至于如何设置这个背景View，当然又要依赖UICollectionView的delegate了。考虑到Section背景应该要关联到Section数据本身，我们应该将设置方法分发到SectionController中，这样就要求delegate要能引用ListAdapter实例。 这时候我们有两种选择：1、在ListAdapter上添加扩展。这样我们可以直接让ListAdapter遵循协议，很适合配合第三方UICollectionViewLayout使用。但使用这种方法时，我们就不能在ListAdapter上设置其他的代理。2、在ListAdapter的两个代理之一上设置一个变量，同时该变量能引用ListAdapter本身。 当然由于IGListKit是用OC编写的，我们总是可以向runtime寻求帮助，但在本文中就不展开讨论了。 一个可能的实现如下：

class Layout: UICollectionViewFlowLayout {
  override func layoutAttributesForDecorationView(ofKind elementKind: String, at indexPath: IndexPath) -> UICollectionViewLayoutAttributes? {
    let superLayoutAttributes = super.layoutAttributesForDecorationView(ofKind: elementKind, at: indexPath)
    guard let delegate = collectionView?.delegate, delegate.responds(to: #selector(Delegate.layoutAttributesForDecorationView(ofKind:at:overriding:))) else { return superLayoutAttributes }
    return (delegate as AnyObject).layoutAttributesForDecorationView(ofKind: elementKind, at: indexPath, overriding: superLayoutAttributes)
  }
}

class Delegate : NSObject, UICollectionViewDelegate {
  weak var adapter: ListAdapter?
  @objc func layoutAttributesForDecorationView(ofKind elementKind: String, at indexPath: IndexPath, overriding layoutAttributes: UICollectionViewLayoutAttributes?) -> UICollectionViewLayoutAttributes? {
    return //从section controller中获得背景数据
  }
}

动画

iOS的动画一直是最令开发者满意的特性之一，在UICollectionView上的表现也同样出色。我们只要将数据源的增加，删除，更新和移动变化告知UICollectionView，它就会自动帮我们配置相应的关键帧动画。 但问题是，数据源中包含各种不同结构的项目，我们怎么才能计算出数据的变化呢？ IGListKit内置了一个diff算法, 任何类型只要继承了ListDiffable 协议，就可以使用IGListDiffXXX相关函数计算出前后两个序列之间的相互关系。同时IGListKit要求所有的view model都实现ListDiffable协议，这样在主线程任何地方，只要ListAdapterDatasource中的内容被修改，我们就可以调用ListAdapter上的performUpdates(animated:completion)方法，来自动补全对应动画。ListDiffable协议的定义如下

public protocol ListDiffable: NSObjectProtocol {
  public func diffIdentifier() -> NSObjectProtocol
  public func isEqual(toDiffableObject object: ListDiffable?) -> Bool
}

很容易就可以发现，这几乎就是Hashable协议的翻版：

public protocol Hashable : Equatable {
  public var hashValue: Int { get }
  public static func ==(lhs: Self, rhs: Self) -> Bool
}

很自然地，我们可以猜测到IGListKit在Diff算法内部应该有一个查找表，以diffIdentifier为键关联新旧数据。但假如键有冲突，Diff的行为会是怎样呢？带着疑问我们来看一下代码：

//IGListDiff.mm
static id IGListDiffing(...
                        NSArray<id<IGListDiffable>> *oldArray,
                        NSArray<id<IGListDiffable>> *newArray,
                        ...) {
  ...
  //这应该就是之前提到的查找表了
  unordered_map<id<NSObject>, IGListEntry, IGListHashID, IGListEqualID> table;
  ...

  vector<IGListRecord> oldResultsArray(oldArray.count);
  //降序遍历
  for (NSInteger i = oldCount - 1; i >= 0; i--) {
      id<NSObject> key = IGListTableKey(oldArray[i]);
      IGListEntry &entry = table[key];
      //记录下数据在旧序列中的位置，由于是降序遍历，靠前的位置会较靠近栈顶
      entry.oldIndexes.push(i);
  }

  //升序遍历新序列，逐个关联旧序列中的位置
  for (NSInteger i = 0; i < newCount; i++) {
    ...
  }
  ...
}

所以当不同项目返回了相同的diffIdentifier时，IGListKit会从序列初始端开始逐个匹配，到末端仍未匹配上的项目会被标记为新插入或是已删除。 在实际应用场景中，大多数的列表都会将新项目插入到整个列表的顶部，如果这时候多个项目返回了相同的diffIdentifier，将会导致这一系列项目的动画计算错误。要想绕过这个问题也很简单，我们只要继承ListSectionController，在didUpdate(to:)方法中，将数据源反序输入diff算法，再将返回结果反序并据此处理动画即可。

总结：

IGListKit很好地解决了UICollectionView中数据源类型的横向拓展问题，同时还额外地管理了过渡动画和cell类型注册。此外，作为一个用Objective-C编写的框架，IGListKit也在命名和文档等方面为适应Swift做了很多工作。但不论怎么说，在现有的UI组件上进行再度封装，总会不可避免地带来一些不便。这一点在应用框架前应当仔细权衡。
